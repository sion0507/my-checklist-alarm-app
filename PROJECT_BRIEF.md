# Project Brief: my-checklist-alarm-app

## Project goal

할 일을 자주 까먹는 개인 사용자를 위해, 오늘 해야 할 일을 아침에 정리해 알려주고, 지정 시간에는 확인 알림을 보내며, 밤에는 미완료 할 일을 정리하도록 돕는 개인용 체크리스트 알림 PWA를 만든다.

가장 중요한 성공 기준은 **iPhone에서 잠금화면/시스템 알림으로 실제 알림을 받는 것**이다.

## Target users

- 주 사용자: 방시온 본인
- 사용 환경: iPhone 중심, 필요 시 PC 브라우저에서도 관리
- 사용 목적: 하루 할 일 누락 방지, 시간 지정 업무 리마인드, 미완료 할 일 정리

## Primary workflows

### 1. 할 일 등록

1. 사용자가 앱을 연다.
2. 할 일 제목을 입력한다.
3. 날짜를 선택한다.
4. 필요하면 시간을 지정한다.
5. 필요하면 반복 규칙을 선택한다.
6. 저장한다.

필드 후보:

- 제목: 필수
- 날짜: 필수
- 시간: 선택
- 메모: 선택
- 반복: 없음/매일/매주/매월
- 완료 여부

### 2. 아침 체크인

1. 설정된 아침 시간에 알림이 온다.
2. 알림은 오늘 할 일 개수와 요약을 알려준다.
3. 앱을 열면 오늘 할 일 목록이 보인다.
4. 앱은 “추가로 기록할 일 있어?”를 묻는다.
5. 사용자는 빠르게 할 일을 추가하거나 넘어간다.

아침 알림 시간은 설정 가능해야 한다. 가능하면 알림음/노래 설정도 지원하되, iOS PWA에서 불가능하면 MVP에서는 제외하고 리스크/후속 과제로 남긴다.

### 3. 지정 시간 할 일 확인

1. 할 일에 시간이 설정되어 있으면 해당 시간에 알림을 보낸다.
2. 알림 문구는 “이 할 일 했어?” 형태로 확인을 유도한다.
3. 앱을 열면 해당 할 일을 완료 처리하거나 나중에 처리할 수 있다.

### 4. 저녁 미완료 정리

1. 기본 23:00에 미완료 할 일 알림이 온다.
2. 알림/앱 화면에서 오늘 미완료 할 일 목록을 보여준다.
3. 각 항목에 대해 사용자가 선택한다.
   - 삭제
   - 다른 날짜로 넘기기
   - 그대로 두기/나중에 결정
4. 넘기기를 선택하면 날짜를 고른다.

저녁 리마인드 시간은 설정 가능해야 한다.

### 5. 반복 할 일

반복 규칙은 MVP에 포함한다.

- 매일
- 매주
- 매월

반복 항목은 각 발생일에 오늘 할 일로 나타나야 하며, 완료 처리와 다음 발생일 생성/표시 규칙을 명확히 해야 한다.

### 6. 월간 캘린더 화면

예시 이미지를 기준으로 iPhone 세로 화면에 최적화된 월간 캘린더 뷰를 제공한다.

주요 UI 기준:

- 상단에는 큰 월 제목을 표시한다. 예: `5월`.
- 월 제목 오른쪽에는 보조 액션 영역을 둔다. 예시 이미지처럼 둥근 pill 형태 안에 주요 버튼/더보기 버튼을 배치할 수 있다.
- 월 제목 아래에는 넓은 둥근 사각형 카드 영역을 둘 수 있다. MVP에서는 오늘 요약, 선택 날짜 요약, 아침 체크인/저녁 정리 카드 중 하나를 표시하는 영역으로 사용한다.
- 요일 헤더는 `일 월 화 수 목 금 토` 순서로 표시한다.
- 날짜 숫자는 넓은 간격의 7열 월간 그리드에 표시한다.
- 일요일/공휴일은 빨간색 계열, 토요일은 파란색 계열, 평일은 검은색/짙은 회색 계열로 표시한다.
- 이전/다음 월에 속한 날짜는 흐린 색으로 표시한다.
- 오늘 날짜는 파란 원형 배경 등으로 강조한다.
- 각 날짜 칸에는 해당 날짜의 할 일을 작은 pill/badge 형태로 표시한다.
- 할 일 pill은 연한 파란 배경, 왼쪽 파란 강조선, 회청색 텍스트 스타일을 기본으로 한다.
- 할 일이 여러 개면 날짜 칸 안에 세로로 쌓아 보여준다.
- 날짜 칸에 표시할 수 있는 개수를 넘으면 `+N` 또는 더보기 표시를 사용하고, 날짜 선택 시 전체 목록을 보여준다.
- 공휴일/기념일 텍스트는 빨간색으로 날짜 아래 표시한다.
- 좌우 스와이프로 이전/다음 월로 이동한다.
- 연/월 선택 UI를 제공해 특정 년도/월로 직접 이동할 수 있다.
- 하단에는 둥근 floating tab bar를 둔다. 예시 기준 탭은 `오늘/홈`, `캘린더`, `설정/사용자` 정도로 시작하고, 캘린더 탭 선택 시 가운데 pill 배경으로 강조한다.

캘린더 화면의 목적은 단순 날짜 선택이 아니라, 한 달 동안의 할 일 분포를 한눈에 보고 특정 날짜의 할 일을 빠르게 열람/추가/수정하는 것이다.

### 7. 기본 홈/내비게이션

- 앱의 기본 진입 화면은 `오늘` 화면으로 한다.
- 월간 캘린더는 하단 탭의 `캘린더` 탭에서 접근한다.
- 설정은 하단 탭의 `설정` 탭에서 접근한다.
- 기본 하단 탭 구성은 `오늘`, `캘린더`, `설정`이다.
- 오늘 화면은 오늘 할 일, 아침 체크인 카드, 저녁 미완료 정리 카드처럼 당일 실행에 필요한 정보를 우선 표시한다.

- 실제 푸시 알림 구현 시도
- 오늘 할 일 화면
- 월간 캘린더 화면: 연/월 선택 가능, 각 날짜 칸에 해당 날짜 할 일 표시, 좌우 슬라이드로 이전/다음 월 이동
- 할 일 추가/수정/삭제
- 완료 체크
- 날짜 지정
- 선택적 시간 지정
- 반복: 없음/매일/매주/매월
- 아침 알림 시간 설정

## Confirmed decision log from grill-me interview

1. 실제 iPhone 잠금화면 푸시를 위해 최소 백엔드/스케줄러 사용을 허용한다.
2. 백엔드/스케줄러는 노트북 상시 실행이 아니라 무료 클라우드 기준으로 계획한다. 유료 전환은 MVP 범위에서 금지한다.
3. 무료 티어만 사용하므로 서버는 정시 발송을 시도하되, iOS/네트워크/무료 플랫폼 제한으로 인한 지연/누락 가능성은 리스크로 명시한다.
4. 서버에는 전체 할 일 DB가 아니라 푸시 발송에 필요한 최소 데이터만 저장한다: 알림 id, 발송 시각, 제목/짧은 문구, 알림 타입, 완료/취소 상태.
5. 반복 할 일은 원본 반복 규칙 + 날짜별 예외 저장 방식으로 관리한다.
6. 로컬 데이터가 원본이며, 할 일 변경 시 서버 예약을 즉시 동기화하고 앱 시작 시 재동기화한다.
7. 서버에는 향후 7일치 알림 예약을 만든다.
8. MVP 필수 알림은 아침 오늘 할 일 알림, 시간 지정 할 일 알림, 밤 23:00 미완료 정리 알림 세 가지 모두다.
9. 알림음/노래 설정은 Phase 0에서 iOS PWA 가능성을 확인하고, 불가능하면 MVP에서 제외하거나 제한 표시한다.
10. 저장소는 설정값은 localStorage, 할 일/반복 예외/예약 상태는 IndexedDB를 사용한다.
11. 월간 캘린더 화면은 사용자가 첨부한 예시 이미지 기준으로 구현한다.
12. 기본 진입 화면은 오늘 화면이고, 캘린더는 하단 탭에서 접근한다.
13. 할 일 추가 UX는 상단 빠른 입력 + 상세 모달을 기본으로 한다. 오늘 화면에서는 제목만 빠르게 입력해 오늘 할 일을 추가하고, 날짜/시간/반복/메모는 상세 모달에서 설정한다. 캘린더에서는 날짜 탭 후 해당 날짜 기준으로 추가한다.
14. 오늘 화면은 완료/미완료를 모두 표시하되, 미완료 항목을 상단에 먼저 정렬하고 완료 항목은 체크/흐린 스타일로 하단에 표시한다.
15. 알림을 눌러 앱에 들어오면 관련 할 일을 강조하고 빠른 액션 `완료`, `삭제`, `다른 날짜로 넘기기`를 제공한다.
16. 반복 할 일의 완료/삭제/날짜 변경 기본 동작은 해당 날짜 발생분에만 적용한다. 반복 규칙 전체 수정/삭제는 상세 화면의 별도 액션으로 제공한다.
17. 할 일 상세 모달 MVP 필드는 제목, 날짜, 시간, 반복, 메모, 알림 on/off를 포함한다.
18. 아침 체크인 알림 기본 시간은 08:00이며 설정에서 변경 가능하다.
19. 아침 체크인 카드는 알림 수신 여부와 관계없이 매일 오늘 화면에 표시한다. 사용자가 당일 `확인 완료`하면 그날은 접히거나 숨김 처리한다.
20. 밤 미완료 정리 알림은 미완료 할 일이 있을 때만 발송한다. 앱 안의 저녁 정리 카드는 매일 표시하되, 미완료가 없으면 `오늘 할 일 모두 완료` 같은 완료 메시지를 보여준다.
21. 수동 백업/복원은 MVP에서 제외한다. 로컬 저장 데이터 삭제 리스크를 명시하고, JSON 내보내기/가져오기는 후속 기능으로 남긴다.
22. 테스트 범위는 단위 테스트, E2E 테스트, 수동 iPhone PWA 체크리스트를 모두 포함한다.

## Constraints and risks

### iOS PWA 알림 제약

- iOS 16.4+에서 홈 화면에 설치된 PWA는 Web Push를 지원한다.
- 사용자가 알림 권한을 허용해야 한다.
- 일반 Safari 탭에서는 기대한 방식으로 동작하지 않을 수 있다.
- 정확한 예약 알림은 브라우저 로컬만으로 한계가 있을 수 있다.
- Web Push는 보통 서비스 워커 + Push API + 서버/푸시 구독 저장 구조가 필요하다.

### 로컬 저장의 한계

- 기기 변경/브라우저 데이터 삭제 시 데이터가 사라질 수 있다.
- iPhone Safari/PWA 저장소 정책 영향을 받을 수 있다.
- 백업/복구는 MVP 비목표로 둔다.

### 노래/알림음 설정

- iOS PWA가 시스템 푸시 알림음을 앱에서 자유롭게 커스터마이즈할 수 있는지 불확실하다.
- 가능하면 설정으로 제공하되, 제약이 확인되면 MVP에서 제외한다.

## Candidate stack and tradeoffs

### Option A: 로컬 우선 PWA + Web Push 보조 서버

- Frontend: React/Vite 또는 Vanilla JS PWA
- Storage: IndexedDB 또는 localStorage
- Push: Service Worker + Push API + 최소 백엔드

장점:

- iPhone 실제 푸시 목표에 가장 가까움
- 앱은 여전히 PWA로 유지 가능

단점:

- “로컬 저장만”보다 구조가 복잡함
- 푸시 구독 저장/예약 발송을 위한 백엔드가 필요할 가능성이 큼

### Option B: 완전 로컬 PWA

- Frontend only
- Storage: IndexedDB/localStorage
- Notification API와 앱 내 리마인드 중심

장점:

- 구현이 빠르고 단순함
- 서버가 필요 없음

단점:

- 앱이 닫혀 있거나 잠금화면 상태일 때 정확한 알림 보장이 어려움
- 사용자의 핵심 목표인 “진짜 푸시 알림”을 만족하지 못할 수 있음

### 추천

Planner는 **Option A를 우선 검토**해야 한다. 사용자는 실제 푸시 알림을 핵심 목표로 확정했다. 다만 데이터 자체는 로컬 저장 중심으로 유지하고, 푸시 예약/발송에 필요한 최소 서버만 붙이는 구조를 우선 제안한다.
